Affordance

Hvilken handling en gjenstand signaliserer. Ting som inviterer til handlinger. Dørhåndtaket sier trykk meg ned. Datamusen klikk på meg! Gjøre det åpenbart for brukeren hvordan et produkt kan brukes.
Relasjon mellom objekt og aktørs fysiske egenskaper

  • Real affordance:

    • hva kan en aktør gjøre. Fysiske gjenstander kan gripes.
    • Ofte åpenbart hva vi kan gjøre med dem
    • Intuitivt. Trenger ikke opplæring.

  • Perceived affordance

    • Hva oppfatter en aktør at han kan gjøre
    • Grafisk grensesnitt har ikke real affordance. De er virtuelle.
    • GUI er derfor viktig at er innlysende: er dette en knapp? Kan jeg dra dette elementet?

  • False affordance (feller)

    • Dekor, egentlig
    • Labels som ser ut som en knapp

Consistency

Like elementer bør ha lik funksjon.
Oppførsel bør være konsistent mht en bestemt oppgave: Vi velger et objekt ved å klippe på venstre musetast. Inkonsistens fører til at brukeren må huske unntak.

Constraints

Affordance betinges av ulike typer constraints:

  • Det fysiske

    • Kroppen og gjenstandens anatomi (stolen, brusflasken

  • Det kulturelle

    • Konvensjoner som læres
    • Rødt betyr stopp
    • GUI-knapper som er grået ut = ikke klikkbare

  • Det logiske

    • Resonnement
    • Kontrollelementers plassering etc

Trinnbaserte oppsett er affordance og constraints satt sammen.

Feedback

Feedback fullfører affordance. Der affordance gir info om handlinger som kan gjøres, gir feedback info om at handlingen er gjort.

Det er 3 typer feedback:

  • Visuell
  • Auditiv
  • Taktil

Sammenhengen mellom kontrollelementer (layout og interaksjon) og deres effekt.
Volumkontroll med knapper over hverandre: Den øvre øker volumet, den nedre senker.

Visibility

Kan jeg se det?
Hvis funksjonaliteten er synlig, forstår brukere hva de kan gjøre. Knapper som er gjemt bak en meny er vanskelig å vite at man kan klikke på – hvis de ikke er synlige engang!

Schneiderman 8 Golden Rules

  1. Consistency
  2. Universal usability
  3. Feedback (forskjellig feedback avhengig av action)
  4. Closure (kvittering, oppsummering)
  5. Prevent errors
  6. Permit mulighet for angring/reversal
  7. User in control
  8. Reduce short-term memory load

Visibility of system status

The system should always keep users informed about what is going on, through appropriate feedback within reasonable time.

Match between system and the real world

The system should speak the users' language, with words, phrases and concepts familiar to the user, rather than system-oriented terms. Follow real-world conventions, making information appear in a natural and logical order.

  • Metaforer: iTunes. Har et bibliotek og en butikk. Metaforer.
  • Paint: Bruker malingsbøtte

User control and freedom

Users often choose system functions by mistake and will need a clearly marked "emergency exit" to leave the unwanted state without having to go through an extended dialogue. Support undo and redo.

  • Implementer angreknapp

Consistency and standards

Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform conventions.

Error prevention

Even better than good error messages is a careful design which prevents a problem from occurring in the first place. Either eliminate error-prone conditions or check for them and present users with a confirmation option before they commit to the action.

  • Fleksibilitet

Recognition rather than recall

Minimize the user's memory load by making objects, actions, and options visible. The user should not have to remember information from one part of the dialogue to another. Instructions for use of the system should be visible or easily retrievable whenever appropriate.

  • Kunnskap i verden, ikke i hodet
    • Knowledge in the world, not in the head.

Flexibility and efficiency of use

Accelerators — unseen by the novice user — may often speed up the interaction for the expert user such that the system can cater to both inexperienced and experienced users. Allow users to tailor frequent actions.

  • Gi proffe brukere proff funksjonalitet (shortcuts), uten å ødelegge for vanlige brukere.

Aesthetic and minimalist design

Dialogues should not contain information which is irrelevant or rarely needed. Every extra unit of information in a dialogue competes with the relevant units of information and diminishes their relative visibility.

  • Alt som ikke tjener en hensikt skal bort. Presenter kun nødvendig data.

Help users recognize, diagnose, and recover from errors

Error messages should be expressed in plain language (no codes), precisely indicate the problem, and constructively suggest a solution.

  • Hjelp brukeren å unngå og håndtere feil

Help and documentation

Even though it is better if the system can be used without documentation, it may be necessary to provide help and documentation. Any such information should be easy to search, focused on the user's task, list concrete steps to be carried out, and not be too large.
